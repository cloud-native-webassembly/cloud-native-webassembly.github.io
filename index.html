**业务背景 **

大数据服务中的UDF是云原生场景下一类特殊的业务负载，它把用户逻辑以函数的方式嵌入到大数据业务自身的查询引擎中， 广义上的UDF是指用户自定义的数据处理函数，如在SQL查询语言中用户可以直接通过如下形式自定义函数

CREATE FUNCTION user_function (parameters)
RETURNS data_type AS
BEGIN
    SQL statements
    RETURN value
END;

在扩展场景下数据服务允许用户通过通用的编程语言（Python、java、c语言等等）来定义函数，以此提高服务的扩展性，满足用户个性化查询需求。

def user_function(parameters)
    # do something
    return value

用户定义好UDF之后，可以在自己的查询语句中直接使用

SELECT ca, user_function(cb, cc) from ta where ca > 0

这种查询在云原生环境中执行，一方面需要保证它的高效，另一方面需要保证嵌入代码的安全性（即嵌入的代码不会对业务系统造成破坏，也不会导致数据泄露）。 目前业界主流的方案都是使用基于虚拟化技术的安全容器（如 袋鼠安全容器），但是这种实现往往都存在着虚拟化层和架构上引入的额外开销。

我们希望提供一种比安全容器更轻量级的安全沙箱运行环境，能为大数据UDF类业务在云原生场景下进一步提高弹性，降低资源开销。 本项目通过和大数据业务方以及高校的合作，探索WebAssembly技术在云原生场景的应用，打造技术竞争力。探索包括以下几个方面：

  • 安全性：WebAssembly是否满足云原生场景下的安全诉求
  • 性能： WebAssembly性能是否优于基于安全容器的沙箱方案
  • 标准化：WebAssembly需要一个和外部程序进行安全高效数据交换的标准化接口
  • 兼容性：WebAssembly对于不同语言的支持程度以及对不同CPU架构的兼容

根据我们对大数据业务的调研，当前大数据业务中很大比例的计算资源都投入在UDF（用户自定义函数）的执行上。我们预期在应用基于WebAssembly的安全沙箱之后（架构如下），大数据业务能在很大程度上提高应用性能，降低资源成本。 这种技术的应用可以进一步推广到所有适用于轻量级安全沙箱的场景，如函数计算等业务。

基于共享内存的wasm沙箱高效数据通信

主机与wasm沙箱的通信问题

Wasm基于堆栈虚拟机，拥有自己的线性地址；除此之外，wasm目前仅支持四种数据类型。以上两点导致主机难以与wasm沙箱进行高效通信。 目前主机与wasm沙箱主要采用序列化-反序列化的传输方式，即在主机端先对需要传输的数据使用序列化工具进行序列化，将结果传入沙箱后，再在沙箱内进行反序列化得到原始数据，反之亦然。这种序列化方案凭借良好的通用性，目前在各种数据传输方案中举足轻重。 然而在获得通用性的同时，频繁的序列化和反序列化操作也占用了相当长的时间，导致传输效率低下。以目前流行的protobuf通信协议为例，使用其作为通信工具，向wasm沙箱中传递一个字符串变量并进行反转。如图1所示，其中序列化-反序列的时间占用了大量时间。 图1 数据通信中的时间占比 综上所述，如果将wasm沙箱应用在实际工作中，必须优化和主机之间的数据通信过程。

基于共享内存的数据通信方案

本研究主要以输入语言为C/C++为研究点切入，其他语言原理相同。

Wasm沙箱虽然拥有自己的一套线性地址，但实际使用的还是主机的物理内存，因此二者可以通过共享内存方式进行数据通信。 图2 共享内存 如图2所示，约定wasm线性空间中的开头0-1G为数据段，并将其映射为主机的一段物理地址；主机端将需要传输的数据写到上述物理内存中，这样wasm沙箱端就可以直接在自己的数据段进行数据读写，无需序列化-反序列化操作。 然而上述方案还有一个亟待解决的问题。目前wasm仅支持32位地址，而绝大多数主机为64位地址，这导致某些数据类型所占长度不同，进而影响沙箱内的数据读写。以C/C++中的指针类型为例，在主机的物理内存中每个指针占用8B空间，而在wasm的线性空间中，指针则定义为4B长度，这会导致wasm沙箱在读取指针变量时少计算4B偏移量，读取其后面的数据地址时都会前移4B，发生错误，具体可参考图3。因此，需要对偏移量进行修正才能保证沙箱对数据正常读写。 图3 偏移问题

为了解决上述问题，首先需要了解wasm模块的编译流程。在wamr（wasm-micro-runtime）中共有三种执行方式，其中aot，即本机机器码的运行额外开销最小，因此在此以编译成.aot文件为背景说明。从各种语言（目前主要为C/C++、Rust等）的输入文件到.aot主要经历两个过程、三种文件类型，即先从输入文件编译成.wasm文件，再由.wasm文件编译成.aot文件；而以上两个编译过程中第一个阶段使用LLVM作为编译器实现，第二个阶段使由wamr中的wamrc工具完成。本研究主要涉及LLVM的编译过程，故重点解释。 LLVM由前端和后端组成，编译过程可概述为，不同语言的输入文件由对应的前端编译成中间语言（IR），再由输出文件对应的后端将IR编译为目标文件，整体编译过程可参考图4。

图4 .aot的编译过程 经分析，编译到.wasm文件时已经丢失了所有数据结构相关信息，变为完全对字节流的操作，无法进行偏移量的修正，因此如果希望使沙箱正确读取数据，需要实现一个定制化的LLVM编译器前端，在输入文件编译到.wasm文件的过程中修改结构体成员变量的偏移量。 具体来说，每个结构体成员变量偏移量的计算都不相同，所以应为不同的结构体生成对应的辅助计算函数；此外IR中读取结构体成员变量时是通过结构体起始地址加对应偏移量的方式，也就是GetElementPtr函数，因此对偏移量的修改操作（也就是加n个4B，n为在结构体中，该变量前有多少个指针/long变量）应该随着该语句的生成进行。 本研究通过修改Clang（LLVM的前端之一）实现了偏移量的计算和修改。整体方案可参考图5。 图5 定制化编译器方案 关于辅助函数的生成，首先在clang/lib/CodeGen/CodeGenModule.cpp文件中实现函数EmitStructDroppedFunc，该函数的作用是根据输入的结构体信息生成对应的辅助函数；并且修改了clang/lib/CodeGen/ModuleBuilder.cpp的HandleTagDeclDefinition函数，该函数在每个结构体编译的过程中都会被调用一次，在该函数中调用EmitStructDroppedFunc，为每个结构体生成对应的辅助函数；为了方便辅助函数的查找和调用，在clang/lib/CodeGen/CodeGenTypeCache.h文件中定义存储（insert_dropped_function）和查找函数(get_dropped_function)，并使用STL的unorderd_map哈希表存储，其键是结构体名称，值为对应的辅助函数。 关于调用辅助函数进行偏移量的修正，主要修改了clang/lib/CodeGen/CGBuilder.h文件中的CreateStructGEP函数，该函数的作用是生成上文描述的GetElementPtr语句，在生成该语句前调用辅助函数，对结构体成员变量的偏移量进行修正。 综上所述，为实现高效的wasm沙箱通信，本研究在wasm线性空间中定义了数据段，通过与主机共享内存的方式进行通信；为了修正因字长差异产生的数据读写问题，本研究实现了一个定制化的编译器，在编译过程中对变量偏移量进行修正，保证了wasm沙箱正确读写数据。
